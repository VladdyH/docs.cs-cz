<span data-ttu-id="e1250-103">Abstrakce je typ, který popisuje kontrakt ale neposkytuje úplnou implementaci kontraktu.</span><span class="sxs-lookup"><span data-stu-id="e1250-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="e1250-104">Abstrakce jsou obvykle implementovány jako abstraktní třídy nebo rozhraní a začne s kvalitně definované sady referenční dokumentaci popisující požadovaný sémantiku typy implementace kontraktu.</span><span class="sxs-lookup"><span data-stu-id="e1250-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="e1250-105">Některé z vašich nejdůležitějších abstrakce v rozhraní .NET Framework zahrnují <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, a <xref:System.Object>.</span><span class="sxs-lookup"><span data-stu-id="e1250-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>  
  
 <span data-ttu-id="e1250-106">Architektury můžete rozšířit pomocí implementace konkrétního typu implementujícího typ podporuje smlouvy abstrakci a pomocí rozhraní API náročné (nasazeny na) tento konkrétní typ odběru.</span><span class="sxs-lookup"><span data-stu-id="e1250-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>  
  
 <span data-ttu-id="e1250-107">Smysl a užitečné abstrakce, která dokáže odolat zkušební doba je velmi obtížné navrhnout.</span><span class="sxs-lookup"><span data-stu-id="e1250-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="e1250-108">Hlavní problémy s dokončením je stále správnou sadu členů, častěji a ne méně.</span><span class="sxs-lookup"><span data-stu-id="e1250-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="e1250-109">Pokud abstrakci má příliš mnoho členů, bude obtížné či nemožné i k implementaci.</span><span class="sxs-lookup"><span data-stu-id="e1250-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="e1250-110">Pokud má moc malý počet členů pro funkci přislíbeném, bude v mnoha scénářích zajímavých zbytečná.</span><span class="sxs-lookup"><span data-stu-id="e1250-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>  
  
 <span data-ttu-id="e1250-111">Příliš mnoho odběrů v rámci také negativně ovlivnit využitelnost rozhraní framework.</span><span class="sxs-lookup"><span data-stu-id="e1250-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="e1250-112">Často je poměrně obtížné porozumět abstrakci bez nutnosti porozumět, jak zapadá do větší přehled o konkrétní implementace a rozhraní API fungujícími v odběru.</span><span class="sxs-lookup"><span data-stu-id="e1250-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="e1250-113">Názvy abstrakce a jejich členy jsou také nutně abstract, které často jsou nejasné a unapproachable bez první Principy širšího kontextu jejich použití.</span><span class="sxs-lookup"><span data-stu-id="e1250-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>  
  
 <span data-ttu-id="e1250-114">Abstrakce však poskytují velmi výkonné rozšíření, což často nesmí shodovat s další mechanismy rozšiřitelnosti.</span><span class="sxs-lookup"><span data-stu-id="e1250-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="e1250-115">Jsou základem mnoho vzorech architektury, jako jsou moduly plug-in IOC (Inversion) ovládacího prvku, kanály a tak dále.</span><span class="sxs-lookup"><span data-stu-id="e1250-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="e1250-116">Jsou také velmi důležité pro testovatelnost architektur.</span><span class="sxs-lookup"><span data-stu-id="e1250-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="e1250-117">Dobré abstrakce umožňují zástupných procedur si náročné závislosti za účelem testování částí.</span><span class="sxs-lookup"><span data-stu-id="e1250-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="e1250-118">Stručně řečeno abstrakce zodpovídají za sought-after bohatost moderních architektur objektově orientovaný.</span><span class="sxs-lookup"><span data-stu-id="e1250-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>  
  
 <span data-ttu-id="e1250-119">**X DO NOT** zadejte abstrakce, pokud jsou neotestovali ve vývoji několik rozhraní API využívání abstrakce a konkrétní implementace.</span><span class="sxs-lookup"><span data-stu-id="e1250-119">**X DO NOT** provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>  
  
 <span data-ttu-id="e1250-120">**✓ DO** zvolit pečlivě abstraktní třídy a rozhraní při navrhování abstrakci.</span><span class="sxs-lookup"><span data-stu-id="e1250-120">**✓ DO** choose carefully between an abstract class and an interface when designing an abstraction.</span></span>  
  
 <span data-ttu-id="e1250-121">**✓ CONSIDER** poskytuje odkaz testy pro konkrétní implementace abstrakce.</span><span class="sxs-lookup"><span data-stu-id="e1250-121">**✓ CONSIDER** providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="e1250-122">Tyto testy by měly umožnit uživatelům k ověření, zda jejich implementace správnou implementaci kontraktu.</span><span class="sxs-lookup"><span data-stu-id="e1250-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
